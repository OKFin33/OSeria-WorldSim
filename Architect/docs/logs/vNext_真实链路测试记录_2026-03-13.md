# vNext 真实链路测试记录 2026-03-13

## 1. 测试目的

验证当前 `codex/vNext` 分支上的 Architect vNext 原型，是否真的符合预期的 UX 方向：

- 问题是否开放、自然，不会过早锁死用户意图
- 泡泡是否像“用户可能顺手会点的回答”，而不是系统按钮
- Mirror 是否真的基于前文累积理解
- Landing 是否正常衔接
- `CompileOutput -> FrozenCompilePackage -> Blueprint / SystemPrompt` 是否按预期产出

本记录使用真实本地后端接口跑通，回答由我模拟普通用户口吻，不刻意正式。

## 2. 测试环境

- Workspace: `/Users/okfin3/project/architect-agent-prototype`
- Branch: `codex/vNext`
- Backend: `http://127.0.0.1:8000`
- Frontend 预期地址: `http://localhost:5173/`
- 调试接口: `GET /api/debug/session/{session_id}`
- 测试会话 `session_id`: `287e971442964ed59f8477c092aed612`

## 3. 总体结论

这轮链路 **能跑通**，但 **UX 质量未达标**。

当前主要问题不是“接口坏了”，而是：

1. `InterviewComposer` 仍然会过早收紧问题空间
2. `BubbleComposer` 虽然比旧版好，但仍然偏系统引导
3. `routing_snapshot.confirmed` 长时间为空，导致：
   - `Forge` 不起作用
   - 最终 `SystemPrompt` 只有 core 模板，没有足够世界专属规则
4. `Mirror -> Landing` 的衔接有 bug
5. `Blueprint` 目前质量偏弱，摘要字段重复

**系统首问**

### Turn 1

**我的回答**

> 想要修仙，但别太老套。最好有那种御剑飞行、剑光很冷的感觉，不过世界本身得有规矩，不是纯爽文乱砍。

**系统回声**

> 御剑飞行，剑光很冷，世界有规矩——这听起来像是一个剑修抬头时，看到的不是无边的自由，而是划破云霄的线条里，藏着某种必须低头的秩序。

**系统问题**

> 在这样的世界里，规矩第一次真正让你感到“被压住了”的，是怎样的一幕？是剑光划过时，有人告诉你“这里不能飞”，还是你发现剑再快，也快不过门规里的一行字？

**泡泡**

- 剑光再冷，也冷不过门规那行字。
- 有人告诉我，这里不能飞。
- 那一刻我才明白，规矩比剑更快。

**当轮观察**

- 问题已经开始明显收紧。
- 用户只是表达了“修仙 + 冷剑光 + 有规矩 + 不要纯爽文”，系统立刻把焦点压到“规矩压人”的单一方向上。
- 泡泡与问题高度同质，像是系统替用户总结，不像用户自然接话。

**Dossier 关键状态**

- `world_dossier.world_premise`:
  - 一个修仙世界，围绕冷冽剑光、御剑飞行与森严规矩展开
- `player_dossier.fantasy_vector`:
  - 靠近剑修或御剑者的位置感，但具体起点和目标未定
- `player_dossier.taste_bias`:
  - 偏冷硬、克制，避免纯爽感
- `routing_snapshot.confirmed`:
  - 空
- `routing_snapshot.exploring`:
  - `dim:combat_rules`
  - `dim:power_progression`
  - `dim:command_friction`

## 9. SystemPrompt

### 实际情况

由于：

- `confirmed_dimensions = []`
- `Forge` 没有 tasks

所以最终 `SystemPrompt` 主要由：

- core templates
- 一长串 `emergent_dimensions`
- player calibration

构成。

### 核心问题

- 没有足够世界专属的 forge 规则
- 成品 prompt 结构上成立，但个性化深度不足
- 这和当前 Architect 的真正目标不匹配

- 长时间停留在 `exploring`
- 不敢进入 `confirmed`

这导致：

1. 访谈中段缺少明确推进感
2. 后段 compile 失真
3. forge 链路空转

## 11. 当前问题汇总

### P1

1. **Landing 逻辑有 bug**
   - Mirror confirm 后没有出现真正 landing 问题
   - 仍然显示 Mirror 文本

2. **`confirmed_dimensions` 无法正常形成**
   - 导致 `Forge` 没有任务
   - 最终 `SystemPrompt` 缺少模块化世界专属规则

### P2

3. **问题过早收紧**
   - 从开放画卷太快压缩到局部二选一/场景细化

4. **泡泡仍偏系统引导**
   - 更像系统建议用户怎么答
   - 不够像用户自然会点的回答

5. **访谈会在同一幕画面里打转**
   - 缺少继续打开世界层次的能力

6. **Mirror 更像高压缩复述，而非更高一级的世界回声**

### P3

7. **Blueprint 当前质量弱**
   - 摘要字段重复
   - 标题弱
   - 缺少像“世界蓝图”的感觉

8. **SystemPrompt 成品个性化深度不足**
   - 不是错误
   - 但不符合 vNext 想追的质量上限

---

## 12. 当前判断

当前 vNext 原型已经不再是“接口没接上”的问题，而是更真实的内容质量问题：

- `DossierUpdater` 还不够稳
- `InterviewComposer` 仍然会过早收紧
- `BubbleComposer` 还没真正做到“像用户会点的话”
- compile / forge 之间的维度冻结策略有问题
- landing 节点有明确 bug

换句话说：

**架构已经基本站住，但内容层和 compile 质量还没达到目标。**

## 13. 建议的下一步排查顺序

1. 修 `Mirror -> Landing` bug
2. 修 `RoutingSnapshot -> CompileOutput` 的确认策略
3. 继续放宽 `InterviewComposer` 的前几轮开放性
4. 继续收 `BubbleComposer`，让它更像用户侧候选回答
5. 最后再修 `Blueprint` 摘要质量
